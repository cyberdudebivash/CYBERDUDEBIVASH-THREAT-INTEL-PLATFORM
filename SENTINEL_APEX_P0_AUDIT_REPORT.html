2 · Root Cause Analysis

RC-1 · P0 · BLOCKING

Bootstrap Version Mismatch — Stale Intel Root Cause

Component: scripts/bootstrap_critical_files.py

Problem: The file on the GitHub main branch is v101.0.0. The local workspace has v114.0. The v101.0.0 version takes a snapshot of the existing manifest (2,463 entries) to /tmp/pre_run_manifest.json on every run, then performs a UNION merge — appending new STIX bundles on top of all old entries. The result is the 2,463-entry stale corpus is carried forward into every run.

Evidence: Workflow log #742 shows:

SENTINEL APEX v101.0.0 — CRITICAL FILE BOOTSTRAP
Snapshot saved: 2463 entries from feed_manifest.json → /tmp/pre_run_manifest.json
No STIX bundle files found — using existing manifest state

Why this didn't matter at first: v101.0.0 was adequate when the manifest was small and new entries were easily visible. Once stale entries accumulated to 2,463, new entries were visible only if the dashboard sorted correctly — which it didn't (RC-6).

Fix: Push v114.0 bootstrap_critical_files.py to GitHub. v114.0 reads ONLY data/stix/CDB-APEX-*.json bundles. No UNION merge. No snapshot/restore. Max 500 entries. Deterministic sort.

RC-2 · P0 · BLOCKING

report_url Stripped from Worker Preview Response

Component: workers/intel-gateway/src/index.js — handlePreview()

Problem: The preview handler maps each item to an explicit set of fields (lines 360–381) and report_url was absent from that map. The dashboard receives the preview array and does item.report_url || item.source_url || '' — since report_url was always undefined, every "View Tactical Dossier" link resolved to source_url (an external news article) or nothing.

// OLD — report_url MISSING:
return {
  id, title, severity, description, tags, threat_type, risk_score,
  iocs, ttps, ioc_count, ttp_count, confidence, timestamp, source,
  stix_bundle, kev_present, epss_score, cvss_score
  // ← NO report_url
};

Fix: Added report_url to the map with a deterministic derivation fallback in case the manifest entry is missing it.

RC-3 · P0 · BLOCKING

HTML Report Files Never Generated — All report_url Links 404

Component: scripts/generate_intel_reports.py — not in workflow

Problem: Every advisory carries report_url = /reports/YYYY/MM/<id>.html. These files were never generated and never uploaded. Clicking "View Tactical Dossier" always resulted in a 404. The script generate_intel_reports.py existed but was not called in sentinel-blogger.yml and lacked brand entry filtering.

Fix: Added Stage 3.6 to the workflow (after manifest cleanup, before R2 upload) that calls generate_intel_reports.py. Reports are synced to R2 bucket sentinel-apex-reports and committed to gh-pages via the git sync step. Script updated to v115.0 with BRAND_KEYWORDS filtering.

RC-4 · HIGH

UNION Merge Causes Stale Intel Dominance

Root: Consequence of RC-1. The v101.0.0 bootstrap merges all historical entries with new ones. The sort is timestamp DESC so new entries appear at the top — but when there are 2,463 stale entries all with similar timestamps, only 50 of them might be from the current day. Because the Worker preview returns 10 items, all 10 are new — but the total count shows 2,514, which confuses freshness metrics.

Fix: Resolved by RC-1 fix. v114.0 bootstrap discards everything except live STIX bundles produced in the current run. Max 500 entries.

RC-5 · HIGH

28+ Concurrent Workflows Writing to Manifest

Problem: The concurrency: group: sentinel-data-writer directive in sentinel-blogger.yml only protects that workflow. Other workflows (sovereign, nexus, genesis, cortex, quantum, etc.) each run on their own schedule and write to feed_manifest.json without being in the same concurrency group. A race condition between any two of them corrupts the manifest.

Recommended Fix: Audit all .github/workflows/*.yml files. Any workflow that writes to data/stix/feed_manifest.json must use concurrency: group: sentinel-data-writer. Workflows that do not produce STIX output should be disabled or have their manifest write step removed.

Status: Not fully resolved in this fix session (requires auditing all workflow files). Partially mitigated by R2 being the single canonical source read by the Worker; GitHub manifest writes are metadata-only.

RC-6 · MEDIUM

Default '7d' Time Filter Hides Fresh Entries with Older Timestamps

Problem: index.html line 4236 set currentTimeRange = '7d' as default. STIX entries enriched from CVE advisories often carry the CVE's original publication date as the timestamp (which may be weeks old). On first page load, these entries are filtered out and the dashboard appears empty or stale.

Fix: Changed default to 'all'. The user can narrow to a time range using the filter UI at any time.

RC-7 · LOW

blog_url Legacy Remnant in index.html

Problem: Line 4111 in index.html watchlist renderer still had ||item.blog_url as a third fallback for link resolution. This is a Blogger reference that was supposed to be fully purged in v111.0.

Fix: Removed ||item.blog_url. Link resolution is now strictly item.report_url || item.source_url. If neither exists, the VIEW button is hidden.
