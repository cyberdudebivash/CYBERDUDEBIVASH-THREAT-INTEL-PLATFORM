Backend APIs

P1 HIGH
PASS API Architecture: Well-Structured Modular Design
api/auth.py, api/billing.py, api/health.py, api/enterprise.py, api/onboarding.py, api/rate_limiter.py, api/paywall_filter.py, api/monetization.py all exist with clear separation of concerns. Tier-based access control implemented. Cloudflare Workers gateway (workers/intel-gateway, workers/revenue-engine) provides edge routing.
HIGH GitHub Pages Deployment: Static-Only Constraint
Root Cause: The platform is deployed on GitHub Pages (static hosting). All Python API modules (auth.py, billing.py, etc.) exist in the repository but are not deployed as a live API server. The live API is served by Cloudflare Workers backed by Cloudflare R2 storage — not by a running FastAPI/Flask server.
Impact: /auth/login, /auth/keys, /api/anomalies, /api/predict endpoints rely on the Cloudflare Worker implementation. Any mismatch between Python logic and Worker logic = silent bugs.

Implementation Plan

  1. Document live API architecture clearly: worker → R2 → GitHub Pages pipeline
  2. Deploy Python API to Railway/Render/Fly.io as the authoritative backend for auth, billing, and subscription management
  3. Cloudflare Worker as CDN/cache layer only — proxies to Railway backend for authenticated calls
  4. Contract test Worker vs Python API on every deploy to prevent behavioral drift
HIGH API Versioning: No /v1/ Prefix on Endpoints
Root Cause: All endpoints served under /api/ without version prefix. Any breaking change breaks all current API consumers with no migration path.
Impact: Enterprise customers integrating with STIX feed or IOC APIs will break silently on any schema change. Enterprise SLAs require API version guarantees.

Implementation Plan

  1. Introduce /api/v1/ prefix for all authenticated endpoints immediately
  2. /api/feed.json maintained as legacy alias for backward compat (301 redirect to /api/v1/feed.json)
  3. Add X-API-Version response header to every response
  4. Sunset policy: v1 supported for minimum 12 months after v2 release

Authentication Systems

P1 HIGH
PASS JWT Auth + API Key System: Cryptographically Sound
auth.py uses secrets.token_hex() for key generation, hmac.compare_digest() for constant-time comparison (no timing attacks), atomic file writes for key store, and tier-based endpoint ACLs. CDB_JWT_SECRET is stored in GitHub Actions secrets (confirmed in CI logs).
CRITICAL No Email Verification — Anyone Can Register as Enterprise
Root Cause: dashboard.html login accepts any email + name and immediately provisions a FREE tier JWT. No email OTP, no domain verification, no business email check.
Impact: Competitors can register, SOC analysts from untrusted orgs get FREE tier tokens, enterprise impersonation possible. No customer identity verification.

Implementation Plan

  1. Add email OTP verification on first registration (send 6-digit code via transactional email)
  2. Business email check for PRO/Enterprise tiers (reject free domains: gmail, yahoo, hotmail for paid plans)
  3. Rate limit registration: Max 3 new accounts per IP per hour
  4. Magic link auth (passwordless, email-based) as primary auth method — more secure than password
  5. Session tokens: 15-minute JWT access tokens + 30-day refresh tokens
HIGH API Key Storage: JSON File on Disk — Not Production Grade
Root Cause: api_keys.json stored in data/auth/ directory. No encryption at rest, no distributed storage, no concurrent-write safety beyond file locks. On GitHub Pages (static), this file is committed to the repository.
Impact: If repository is public, API keys are exposed. Race conditions possible under concurrent writes. Single point of failure.

Implementation Plan

  1. Migrate to Cloudflare KV for API key storage (already have CF_ACCOUNT_ID secret)
  2. Store only hashed keys in KV — never plaintext. SHA-256(key) as lookup hash
  3. Key metadata (tier, created_at, usage) in KV value as JSON
  4. Key rotation endpoint: POST /auth/keys/:id/rotate — atomic KV update
  5. Audit log every key event to append-only Cloudflare D1 or R2 JSONL

API Security

P1 HIGH
PASS Security Headers: Production Grade (_headers)
HSTS with preload, CSP with strict directives, X-Frame-Options: DENY, CORP, COOP, Permissions-Policy denying camera/mic/geolocation/payment. All OWASP Secure Headers Project v4.0 validated. Separate cache policies for HTML vs API vs Reports. ✅
HIGH CORS: Access-Control-Allow-Origin: * on /api/*
Root Cause: _headers sets Access-Control-Allow-Origin: * for all /api/* endpoints. Any website can call the API on behalf of any authenticated user.
Impact: CSRF-style attacks where a malicious site silently consumes a user's API quota. Credentials-with-wildcard attack vector.

Implementation Plan

  1. Replace wildcard with allowlist: intel.cyberdudebivash.com, cyberdudebivash.com, www.cyberdudebivash.in
  2. For public /api/feed.json: Wildcard acceptable (public data)
  3. For authenticated endpoints: CORS must reflect the requesting origin with credentials
  4. Add CSRF token to all state-changing POST/DELETE requests
CRITICAL No WAF — No DDoS Protection Beyond GitHub Pages CDN
Root Cause: Platform relies on GitHub Pages CDN for delivery. No Cloudflare WAF rules, no rate-limit-at-edge for malicious actors, no bot protection.
Impact: A targeted DDoS against intel.cyberdudebivash.com during a high-profile CVE release would take down service for enterprise customers. Enterprise SLA cannot be guaranteed without WAF.

Implementation Plan

  1. Route domain through Cloudflare proxy (already using CF) — enable Orange Cloud
  2. Enable Cloudflare WAF free tier: OWASP ruleset, SQL injection protection, XSS protection
  3. Rate limiting at edge: 100 req/min per IP on /api/* endpoints
  4. Bot Fight Mode: Enable to block scrapers
  5. Under Attack Mode: Enable during active DDoS incidents
