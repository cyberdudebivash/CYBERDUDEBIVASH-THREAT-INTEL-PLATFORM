## POLICY STATEMENT

All threat intelligence HTML reports generated by the SENTINEL APEX pipeline are
**AUTHORITATIVE DEPLOYMENT STATE**. They are not ephemeral runtime artifacts.
Once generated and committed, a report MUST remain accessible at its canonical URL
for the lifetime of the platform.

Historical reports MUST NEVER disappear due to:
- Git stash recovery race conditions
- Shallow checkout (`fetch-depth: 1`) losing ORIG_HEAD
- Deployment artifact gaps (folder: . pollution)
- CDN propagation failures
- gh-pages branch corruption

## REPORT LIFECYCLE GOVERNANCE

### 1. Report Generation (Authoritative State Creation)

**Stage:** 3.2 (report_generator.py)

**Governance rules:**
- A report is AUTHORITATIVE once written to `reports/YYYY/MM/intel--{id}.html`
- God Mode MUST protect existing reports ≥ 60KB from overwrite
- God Mode MUST regenerate if `report_url` public path is missing (FIX-3, v154.0)
- Every generated report MUST have a corresponding `report_url` in the manifest

**Retention policy:** PERMANENT — no report may be deleted after creation.

### 2. Report Persistence in Git (Committed State)

**Stage:** 4 (safe_git_commit.py)

**Governance rules:**
- All `reports/**/*.html` files MUST be committed to `main` branch
- Reports are NOT gitignored (`.gitignore` does not exclude `reports/`)
- `safe_git_commit.py` MUST include a pre-commit count check for reports
- ORIG_HEAD recovery guard MUST fire if reset --hard wipes any reports
- `fetch-depth: 0` ensures ORIG_HEAD is always resolvable

**Integrity verification:**
```python
# safe_git_commit.py MUST log:
[pre-commit] reports/ has N HTML report(s)
[post-commit] reports/ has N HTML report(s) — INVARIANT CONFIRMED
```

### 3. Report Inclusion in Dist Artifact (Deployment State)

**Stage:** 5.4.6 (build_dist_artifact.py)

**Governance rules:**
- `dist/reports/` MUST contain ALL reports from `reports/` in the working tree
- No report present in `reports/` may be excluded from `dist/reports/`
- `dist/deployment_manifest.json` MUST list every report with SHA-256 checksum
- If any `report_url` path is missing from `dist/`, build HARD FAILS

**Invariant:**
```
|reports/ on disk| == |dist/reports/| == |deployment_manifest.json report count|
```

### 4. Report Verification Pre-Deploy (Governance Gate)

**Stage:** 5.4.7 (dist_artifact_verifier.py)

**Governance rules:**
- 10-check battery MUST pass before deployment proceeds
- SHA-256 checksums MUST match manifest (spot-check sampling)
- Zero zero-byte report files permitted
- Report count MUST be >= 10 (minimum viable deployment)
- ALL `report_url` feed paths MUST exist in `dist/`

**Gate contract:** Pages deployment is BLOCKED if any verifier check fails.

### 5. Report Deployment to gh-pages (Published State)

**Stage:** 5 (GitHub Pages Deploy)

**Governance rules:**
- Deployment MUST use `folder: dist` (governed artifact only)
- `clean: true` ensures gh-pages exactly mirrors `dist/` (no stale files)
- `dist/reports/` contains the complete report corpus — historical preservation is
  guaranteed because `dist/reports/` is built from the full working tree
- Every report in `dist/` becomes accessible at its canonical URL

**Canonical URL contract:**
```
dist/reports/YYYY/MM/intel--{id}.html
  → https://intel.cyberdudebivash.com/reports/YYYY/MM/intel--{id}.html
```

**Stage:** 5.8.1b (report_url_canary.py)

## HISTORICAL REPORT PRESERVATION MECHANISM

Historical reports are preserved through three complementary mechanisms:

**1. Git tracking:** All reports are committed to `main`. With `fetch-depth: 0`, every
   historical commit is fetchable. Reports can be recovered from any prior commit.

**2. ORIG_HEAD restore:** When `git reset --hard` runs (stash recovery in safe_git_commit.py),
   ORIG_HEAD points to the pre-reset commit. Reports lost by the reset are recovered via:
   ```
   git checkout ORIG_HEAD -- reports/
   ```
   This requires `fetch-depth: 0` to resolve ORIG_HEAD.

**3. Dist/ full corpus:** `build_dist_artifact.py` copies the ENTIRE `reports/` tree
   (not just new reports). With `clean: true` on Pages deploy, gh-pages always
   reflects the current complete report corpus. Historical reports from 2019-2026
   are deployed on every run.

## REPORT CONTINUITY FAILURE PLAYBOOK

If a report URL becomes 404 after deployment:

1. **Identify:** Check `dist/deployment_manifest.json` for the report ID
2. **Diagnose:** Confirm file exists in `reports/` on runner (check Stage 3.2 logs)
3. **Verify dist/:** Check Stage 5.4.6 log for any MISSING entries
4. **Check gh-pages:** `git ls-files --with-tree=origin/gh-pages | grep {id}`
5. **Recover:** Run pipeline with `force_full_sync: true` — this bypasses God Mode
   and regenerates all reports, rebuilds dist/, and redeployes
6. **Escalate:** If reports are missing from git history entirely, restore from the
   last known-good commit: `git checkout {known_good_sha} -- reports/`
