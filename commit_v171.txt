fix(chrome-desktop): v171 RC10+RC11 — DEFINITIVE Chrome blank threat map fix

ROOT CAUSES ELIMINATED (RC10 + RC11):

RC10: overflow:hidden on #cdb-threat-map-panel — Chrome D3D11/ANGLE stencil-clip trap
  Chrome's D3D11/ANGLE compositor creates a stencil clipping region for any element
  with overflow:hidden. When the absolutely-positioned canvas was GPU-promoted to a
  separate compositor layer (RC11/translateZ), Chrome clipped that compositor layer
  using the parent's stencil region — resulting in a BLANK compositor layer, i.e.,
  a blank canvas. Edge does NOT apply the overflow stencil to separate compositor
  layers in this manner; hence Edge rendered correctly and Chrome did not.
  Fix: overflow:visible !important on #cdb-threat-map-panel. Panel has explicit
  height:340px — nothing can overflow. Stencil eliminated. Compositor layer unclipped.

RC11: promote()/translateZ(0) call in boot() — GPU layer promotion inside stencil-clip
  The CDB_COMPOSITOR.promote() call (and its inline translateZ(0) fallback) in boot()
  added transform:translateZ(0) !important + will-change:transform !important to the
  canvas AFTER first paint. This GPU layer promotion, combined with the overflow:hidden
  stencil-clip (RC10), was the exact mechanism that created the blank compositor layer.
  Canvas2D rendering does NOT require GPU layer promotion for correctness — it only
  benefits scroll performance, which is irrelevant here. Eliminated entirely.
  Replaced with a clean double-RAF compositor stability buffer (no promote, no translateZ).
