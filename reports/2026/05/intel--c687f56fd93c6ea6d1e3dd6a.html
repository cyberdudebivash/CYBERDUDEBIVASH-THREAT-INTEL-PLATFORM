Executive Summary

CRITICAL — EMERGENCY PATCH REQUIRED WITHIN 24 HOURS
Unauthenticated SQL injection grants full database access. All stored LLM provider API keys (OpenAI, Anthropic, Google, AWS Bedrock) must be treated as compromised on unpatched deployments.

CVE-2026-42208 is a CRITICAL severity SQL injection vulnerability in LiteLLM — an open-source Python library and proxy server widely used to route, load-balance, and unify requests across multiple Large Language Model (LLM) providers including OpenAI, Anthropic, Google Gemini, AWS Bedrock, Azure OpenAI, and Cohere. LiteLLM proxy is deployed extensively by AI startups, enterprise AI teams, and ML platform engineers to manage multi-provider LLM access under a single API gateway.

The vulnerability exists in LiteLLM proxy's API key verification flow. When the proxy validates incoming API keys against its backend database, the key value is passed directly into a SQL query without proper parameterised binding (prepared statements). An attacker can supply a specially crafted API key string containing SQL metacharacters to manipulate the query logic, resulting in:

  • Complete authentication bypass — attacker gains unrestricted access to all LLM endpoints without a valid API key
  • Full database read access — extraction of all stored API keys, user records, usage logs, and LLM provider credentials via UNION SELECT injection
  • LLM provider credential theft — stolen OpenAI/Anthropic/Google keys enable direct financial abuse of your LLM quota ($10K–$500K+ exposure per provider)
  • Secondary credential exfiltration — database may contain downstream application secrets, internal tokens, and OAuth credentials

Exploitation requires only a valid HTTP client and knowledge of a standard LiteLLM API endpoint path (/v1/chat/completions, /v1/embeddings, etc.). No prior credentials, no special tooling, no user interaction. Fully automatable mass exploitation.

Section 05

Technical Analysis — Deep Dive

Root Cause: Unsanitised SQL Query Construction in API Key Verification

LiteLLM proxy maintains a local SQLite/PostgreSQL database containing API keys, user records, usage budgets, and LLM provider credentials. When an incoming API request presents a key via the Authorization: Bearer or x-api-key header, the proxy validates it against this database before routing the LLM request.

The vulnerability resides in the key verification function. The affected code constructs the SQL query using string concatenation of the raw user-supplied API key value rather than parameterised queries (prepared statements). The vulnerable pattern is approximately:

# VULNERABLE CODE — string concatenation (simplified representation)
query = f"SELECT * FROM api_keys WHERE key_value = '{api_key}' AND is_active = 1"
result = db.execute(query)

# SECURE PATTERN — parameterised query (should be)
query = "SELECT * FROM api_keys WHERE key_value = ? AND is_active = 1"
result = db.execute(query, (api_key,))

Attack Payloads — Working SQL Injection Strings

An attacker supplies one of the following values as the API key, causing the SQL query to return all rows (bypassing authentication) or extracting sensitive data:

# Authentication bypass — OR condition always true
Authorization: Bearer ' OR '1'='1
Authorization: Bearer ' OR 1=1--
Authorization: Bearer ' OR 'x'='x
x-api-key: anything' OR '1'='1'--

# Data extraction — UNION SELECT to dump all API keys
Authorization: Bearer ' UNION SELECT key_value,1,1,1 FROM api_keys--

# Extract LLM provider credentials table
Authorization: Bearer ' UNION SELECT provider_key,provider_name,1,1 FROM llm_providers--

Exploitation Impact Cascade

  1. Phase 1 — Auth Bypass: Attacker gains unrestricted access to all LiteLLM proxy endpoints (/v1/chat/completions, /v1/embeddings, /v1/images/generations) without a valid key. All configured LLM models now accessible.
  2. Phase 2 — LLM API Abuse: Attacker routes high-volume, expensive LLM requests through the proxy, consuming the victim organisation's OpenAI/Anthropic/Google quota. A single attacker session can burn $10,000–$100,000 in API credits in hours.
  3. Phase 3 — Credential Extraction: Via UNION SELECT injections, attacker extracts: all proxy API keys (to share with others or sell), LLM provider API keys (OpenAI sk-*, Anthropic sk-ant-*, Google AIzaSy*), user email records, usage budget data, and any additional secrets stored in the proxy database.
  4. Phase 4 — Secondary Breach: Extracted LLM provider keys are used to access the upstream LLM provider platforms directly, potentially exposing prompt history, fine-tuning data, and stored conversations depending on provider settings.

Detection & Response Playbook — CRITICAL Priority

IMMEDIATE — 0 to 2 Hours (EMERGENCY)
  • Take LiteLLM proxy offline or behind VPN immediately if internet-exposed. Do not wait for patch — service continuity is secondary to preventing credential theft.
  • Deploy WAF rule blocking SQL metacharacters (', ", --, UNION, SELECT, OR 1=1) in Authorization/x-api-key headers at all ingress points.
  • Rotate ALL LiteLLM database credentials immediately: LLM provider API keys (OpenAI, Anthropic, Google, AWS Bedrock, Azure), proxy API keys, database passwords.
  • Check LLM provider dashboards (OpenAI Usage, Anthropic Console, Google AI Studio) for anomalous spend spikes in the last 30 days — an unexpected cost spike is a post-exploitation indicator.
  • Pull last 30 days of LiteLLM access logs and search for: SQL metacharacters in auth headers, HTTP 200 responses with anomalous key values, rapid sequential requests from single IPs.
Short-Term — 2 to 24 Hours
  • Apply LiteLLM patch: pip install --upgrade litellm to latest patched version. Validate by testing SQLi payload against /v1/chat/completions — expect 401 Unauthorized.
  • Deploy APEX Sigma rule (Section 18) into Splunk/Elastic/Sentinel with 30-day retro-hunt. Focus on Authorization header values containing ', --, UNION.
  • Re-issue all proxy API keys to authorised users with new key values. Invalidate all previously issued keys in database.
  • Review LLM provider account access logs for logins or key usage from unknown IPs or geolocations — stolen key usage may have already begun.
  • Notify LLM providers (OpenAI, Anthropic, Google) if keys are confirmed compromised — request emergency key invalidation and usage investigation.
Medium-Term — 1 to 7 Days
  • Migrate LiteLLM database to parameterised queries across all SQL-touching code paths — review full codebase for additional unsanitised input handling.
  • Implement LLM API key vault (HashiCorp Vault, AWS Secrets Manager) rather than storing provider keys in LiteLLM database plaintext.
  • Add LiteLLM proxy to CMDB vulnerability tracking — enforce automated patch alerting for future litellm package releases.
  • Assess regulatory notification obligations: if personal data (user emails, usage data) was accessible via exploitation window, evaluate GDPR/DPDP/HIPAA disclosure requirements.
  • Quantify financial exposure: calculate LLM API spend anomaly during potential exploitation window; prepare cyber-insurance notification if losses exceed policy threshold.
