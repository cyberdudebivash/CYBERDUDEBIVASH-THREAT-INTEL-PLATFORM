## Program Overview

The SENTINEL APEX MSSP Partner Program enables Managed Security Service Providers to deliver production-grade Cyber Threat Intelligence to their customer base under a white-label or co-branded model. Partners receive bulk-priced access to the SENTINEL APEX feed, sub-tenant management infrastructure, and technical integration support.

The program is designed for MSSPs who:
- Operate SOC environments serving 5–100+ customer organizations
- Require machine-readable CTI feeds for SIEM enrichment, detection engineering, or customer reporting
- Want to deliver branded threat intelligence without operating their own feed infrastructure

## Section 4: Sub-Tenant Management

### 4.1 Sub-Tenant Architecture
Each MSSP partner receives a master MSSP API key. Sub-tenants are created and managed under the partner's account:

```
MSSP Master Account
├── Sub-Tenant: Customer A (key: apex_xxxxx, tier: pro)
├── Sub-Tenant: Customer B (key: apex_yyyyy, tier: enterprise)
├── Sub-Tenant: Customer C (key: apex_zzzzz, tier: pro)
└── ...up to 100 sub-tenants
```

### 4.2 Sub-Tenant API
Partners can programmatically manage sub-tenants via the MSSP management API:

```bash
# Create a sub-tenant key
curl -X POST https://intel.cyberdudebivash.com/api/mssp/tenants \
  -H "Authorization: Bearer YOUR_MSSP_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Customer A",
    "tier": "pro",
    "rate_limit": "standard",
    "metadata": {"industry": "financial", "region": "us-east"}
  }'

# List all sub-tenants
curl https://intel.cyberdudebivash.com/api/mssp/tenants \
  -H "Authorization: Bearer YOUR_MSSP_KEY"

# Get sub-tenant usage stats
curl https://intel.cyberdudebivash.com/api/mssp/tenants/{tenant_id}/usage \
  -H "Authorization: Bearer YOUR_MSSP_KEY"
```

### 4.3 Sub-Tenant Isolation
Sub-tenant keys are logically isolated:
- Rate limits are enforced per sub-tenant key (not pooled)
- API key compromise in one sub-tenant does not affect others
- Partner can revoke any sub-tenant key independently
- Sub-tenant keys do not have MSSP-tier API access — they are capped at their assigned tier

### 4.4 Sub-Tenant Reporting
MSSP partners receive a monthly usage report per sub-tenant showing: API calls, feed pulls, rate limit events, and tier. This supports partner billing of their end customers.

### 7.2 Sub-Tenant Key Lifecycle Management

```python
def provision_sub_tenant(mssp_key, customer_name, tier="pro"):
    """Provision a new sub-tenant API key."""
    response = requests.post(
        f"{BASE_URL}/api/mssp/tenants",
        headers={"Authorization": f"Bearer {mssp_key}"},
        json={"name": customer_name, "tier": tier}
    )
    data = response.json()
    return data.get("api_key"), data.get("tenant_id")

def revoke_sub_tenant(mssp_key, tenant_id):
    """Revoke a sub-tenant key upon contract termination."""
    response = requests.delete(
        f"{BASE_URL}/api/mssp/tenants/{tenant_id}",
        headers={"Authorization": f"Bearer {mssp_key}"}
    )
    return response.status_code == 200
```

### 7.3 SIEM Push Architecture for Multi-Tenant SOC
Recommended architecture for MSSPs operating a centralized SIEM:

```
SENTINEL APEX Feed (every 15min)
    ↓
MSSP Feed Aggregator (Cloudflare Worker or Python service)
    ↓
Enrichment: add tenant_id, industry_tag, alert_routing
    ↓
Multi-tenant SIEM (Splunk ES, Microsoft Sentinel, Elastic SIEM)
    ↓
Per-tenant detection rules → per-tenant alert queues → SOC analyst
```
